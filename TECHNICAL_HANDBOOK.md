# PX4 ROS 2 Workspace Technical Handbook

## Purpose

This workspace combines PX4 Autopilot, ROS 2, and LiDAR-based localization tooling for two main use cases:

1. Indoor GPS-denied localization and external-vision feedback for PX4.
2. PX4 SITL simulation with keyboard or joystick teleoperation.

The workspace is a colcon-based ROS 2 workspace. The source code lives in `src/`, while `build/`, `install/`, and `log/` are generated by colcon and should be treated as build artifacts.

## Architecture Overview

### 1. Localization and external vision

The localization stack is designed for indoor or GPS-denied operation. The flow is:

1. PX4 publishes vehicle odometry through ROS 2.
2. `odom_bridge_node` converts PX4 odometry into ROS 2 odometry and a TF tree rooted at `odom`.
3. `rf2o_laser_odometry` uses `/scan` to estimate short-term motion and publishes the dynamic `odom -> base_link` transform.
4. `slam_toolbox` consumes the scan stream and TF tree to build a map and publish `map -> odom`.
5. `vision_odom_bridge` reads the corrected pose from TF and sends it back to PX4 as external vision odometry.

This arrangement lets PX4 fuse a LiDAR-corrected pose instead of relying only on optical flow or another brittle horizontal aid source.

### 2. Teleoperation and SITL

The teleoperation stack is for simulation and remote control. The flow is:

1. `px4_bringup` starts PX4 SITL, Gazebo, the Micro XRCE-DDS agent, and supporting bridges.
2. `px4_offboard` starts an offboard controller node.
3. Keyboard or joystick input is converted into offboard control commands.
4. PX4 receives those commands and flies the simulated vehicle.

## How The Localization Stack Works

The localization stack is controlled by [src/odom_bridge/launch/full_stack_launch.py](src/odom_bridge/launch/full_stack_launch.py).

It launches four stages in order:

* `odom_bridge_node` starts first and publishes `/odom` and the static `base_link -> lidar` transform.
* `rf2o_laser_odometry` starts after a delay and owns the `odom -> base_link` TF edge.
* `slam_toolbox` starts next and publishes `map -> odom`.
* `vision_odom_bridge` starts last and pushes the corrected pose back into PX4.

The launch file uses staggered startup delays so each stage has the upstream data it needs before it begins matching.

Important implementation detail:

* The PX4 odom bridge must not publish the dynamic `odom -> base_link` transform if rf2o is running, or the TF tree will conflict.
* The code in [src/odom_bridge/odom_bridge/odom_bridge_node.py](src/odom_bridge/odom_bridge/odom_bridge_node.py) is responsible for the PX4-to-ROS odometry conversion and static LiDAR transform.
* The code in [src/odom_bridge/odom_bridge/vision_odom_bridge.py](src/odom_bridge/odom_bridge/vision_odom_bridge.py) converts the corrected pose back into PX4 external vision odometry.

## How The Teleoperation Stack Works

The teleoperation stack is centered on [src/ROS2-PX4_Drone_Teleoperation_Using_Joystick/px4_bringup/launch/minimal.launch.py](src/ROS2-PX4_Drone_Teleoperation_Using_Joystick/px4_bringup/launch/minimal.launch.py).

It brings up the full simulation environment, then you launch a control mode from `px4_offboard`:

* [keyboard_teleop.launch.py](src/ROS2-PX4_Drone_Teleoperation_Using_Joystick/px4_offboard/launch/keyboard_teleop.launch.py) for keyboard input.
* [joystick_teleop.launch.py](src/ROS2-PX4_Drone_Teleoperation_Using_Joystick/px4_offboard/launch/joystick_teleop.launch.py) for joystick input.

The offboard package contains the actual controller and teleop nodes used by those launches.

## Main Packages And Their Roles

* `odom_bridge`: PX4 odometry conversion, TF publishing, and external vision feedback.
* `rf2o_laser_odometry`: scan-matching odometry from planar laser scans.
* `slam_toolbox`: map-building and drift correction.
* `px4_msgs`: PX4 ROS 2 message and service definitions.
* `px4_ros_com`: ROS 2 / PX4 bridge examples and frame transform helpers.
* `px4_bringup`: sim bringup for PX4 SITL and supporting services.
* `px4_offboard`: keyboard, joystick, and velocity-based offboard control.

## How To Use It

### Environment expectations

This workspace is set up for Ubuntu 22.04 with ROS 2 Humble, though some packages may also be usable on other ROS 2 distributions if the dependencies are available.

Before running anything, source your ROS 2 environment and build the workspace:

```bash
cd /home/azwad/px4_ros2_ws
colcon build
source install/setup.bash
```

If you are building or debugging a specific subset, you can also build just the relevant packages.

### Run the localization stack

Use this when you want LiDAR-anchored pose estimation and PX4 external vision feedback:

```bash
cd /home/azwad/px4_ros2_ws
colcon build --packages-select odom_bridge rf2o_laser_odometry
source install/setup.bash
ros2 launch odom_bridge full_stack_launch.py
```

What to expect:

* `/scan` should be available from the Gazebo LiDAR bridge or your own LiDAR source.
* TF should contain `odom -> base_link` and later `map -> odom`.
* PX4 should receive `/fmu/in/vehicle_visual_odometry` from `vision_odom_bridge`.

### Run the teleoperation demo

Use this when you want a simulated drone you can fly from the keyboard or a joystick:

```bash
cd /home/azwad/px4_ros2_ws
colcon build
source install/setup.bash
ros2 launch px4_bringup minimal.launch.py
ros2 launch px4_offboard keyboard_teleop.launch.py
```

For joystick control, replace the last command with:

```bash
ros2 launch px4_offboard joystick_teleop.launch.py
```

## Data And Frame Conventions

The workspace follows the usual PX4 and ROS 2 frame split:

* PX4 odometry arrives in PX4-native conventions.
* ROS 2 uses `odom`, `base_link`, and `map`.
* The odom bridge handles conversion between those conventions.
* slam_toolbox introduces the `map -> odom` correction layer.

The localization stack is designed so TF, not raw topic replay, is the source of truth for corrected pose.

## Practical Notes

* `src/odom_bridge/README.md` is the most technically important project-specific document in the workspace.
* The teleop README contains more setup narrative, but the actual package names in the workspace are `px4_bringup` and `px4_offboard`.
* The `install/`, `build/`, and `log/` directories are generated. Do not edit them by hand.
* `vision_odom_bridge.py` is the node that closes the loop back into PX4.

## Suggested Workflow

1. Decide whether you need localization or teleoperation.
2. Build only the packages you need if you are iterating on one stack.
3. Source `install/setup.bash` before launching anything.
4. Start the appropriate launch file.
5. Inspect TF, `/odom`, and PX4 topics in RViz2 or `ros2 topic` when debugging.
